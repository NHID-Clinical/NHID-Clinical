NHID-Clinical: Observations on AI Voice Agents Calling Payers

I spent months working in payer operations, handling eligibility, claims, and authorization calls while protecting PHI. Starting in 2025, our call center began receiving more AI voice agents calling on behalf of provider offices. At first they passed verification, so we treated them like regular calls. Over time, clear problems emerged around disclosure timing, authorization verification, and escalation. This project documents those patterns and proposes a voluntary, testable set of behaviors for AI agents in B2B administrative calls. It is not a standard, not a certification, and does not define compliance requirements. It is an open reference implementation and a conversation starter.

Current state (May 2026): This project is a pre-standardization reference implementation. We have a working event schema, a deterministic policy engine, a machine-readable conformance test suite, and example failure traces. No organization has piloted or adopted these artifacts yet. We are actively seeking feedback from payers, providers, and health IT teams. — Explore the technical reference →

The problem

An AI voice agent calls a payer's office. It sounds professional. It provides provider credentials and asks about a claim. The representative answers — sharing details they would only share in a legitimate workflow.

Three minutes in, the rep gets suspicious: "Wait — are you actually a person?"

Only then does the system admit it's automated.

Operational data has been shared. There's no clear record of when disclosure happened. And there's no way to know whether the credentials were real or scraped from a public directory.

This is impersonation latency ("impersonation latency" — the operational drag when a human can't quickly tell they're talking to an AI). It wastes time, erodes trust, and creates risk on both sides of the call. And it's happening in payer–provider workflows across healthcare daily.

There's a concrete version of this problem that goes beyond late disclosure: an AI can look up any provider's NPI from the public NPPES registry, call a payer, provide that valid NPI, and request claim details. The payer has no real-time way to know whether the caller is actually authorized by that provider. This isn't hypothetical — it's trivially easy with today's voice AI tools. NHID-Clinical aims to name this gap; a cryptographic authorization layer (planned for v1.4) aims to fix it.

Important distinction: NHID-Clinical v1.3 does not verify caller identity or agent authenticity. It standardizes observable disclosure and trace behaviors. The authorization gap is documented — not yet solved.

Payers who ask vendors to adopt NHID-Clinical gain machine‑auditable trace logs for voice agent identity — no certification theatre required.

What NHID-Clinical asks of AI agents

Four behaviors. No certification. No enforcement. Just a written-down baseline.

Behavior What it means
Disclose upfront Say you're automated before asking for any data.
No deceptive audio No fake breathing, filler sounds, or human-name openers.
Human escalation When someone asks to speak to a person, make it immediate.
Basic audit log Record what happened and when. Enough to reconstruct the call.

These are not enforceable requirements. They are a written-down starting point that vendors can reference in contracts and payers can test with the open-source conformance suite.
